# 정보처리산업기사 CBT 모의시험

정보처리산업기사 필기/실기, SQLD 객관식 문제를 실제 CBT(Computer Based Test)처럼 연습할 수 있는 웹 서비스입니다.<br/>
단순 문제풀이 페이지를 넘어, 실제 사용자 피드백을 반영하며 `학습 UX`, `AI 보조 해설`, `운영/분석 대시보드`까지 함께 확장한 서비스형 프로젝트입니다.

## 프로젝트 요약

- 사용자에게는
  - 회차별 CBT / 실기 / SQLD / 오답 재학습 / 랜덤풀이를 제공하는 학습 서비스

- 운영자(나)에게는
  - 신고 처리, GPT 캐시 검수, 유입/완료/합격률 확인이 가능한 관리자 대시보드

- 핵심 포인트
  - 기능 구현에서 끝나지 않고, 실제 사용 데이터를 수집/분석해 개선 우선순위를 정하는 운영 루프까지 구현

## 사용자 피드백 기반으로 성장한 프로젝트 

- 시작점
  - 회차별 CBT 풀이 기능 중심으로 시작

- 사용자가 들어오면서 생긴 요구/문제
  - 해설 이해가 어렵다 / 더 자세한 설명이 필요하다
  - 문제 오류를 빠르게 전달하고 싶다
  - 운영자가 어떤 문제가 자주 이슈가 되는지 한눈에 보기 어렵다
  - 실제 외부 사용자가 얼마나 들어오는지 구분이 어렵다

- 대응하며 확장한 기능
  - GPT 보조 해설 + 캐시 재사용(중복 호출 감소)
  - 문제 신고 기능 + 관리자 신고 처리 UX(그룹화/정렬/선택 삭제)
  - GPT 캐시 조회 탭(재호출 없이 품질 검토)
  - IP 조회(로컬 제외/내 IP 제외/외부 추정 사용자 수)
  - `전체 / 필기 / 실기 / SQLD` 기준 관리자 필터
  - 시험 유형 확장(실기, SQLD)

- 결과적으로 보여줄 수 있는 것
  - “기능 추가”가 아니라, `사용자 반응 → 개선 → 운영 확인`이 반복된 성장형 프로젝트

## 관리자 대시보드

- 아래 내용은 관리자 대시보드의 일자별 리스트(방문/완료/합격률)에서 확인한 데이터입니다.
- 이 섹션에서 보여주고 싶은 점
  - 실제 외부 유입 여부를 확인
  - 단순 방문 수만 보지 않고, `완료`와 `합격률`까지 같이 추적
  - 기능을 늘린 뒤 결과를 대시보드 지표로 검증하는 구조를 만듦

- 데이터 집계
  - `02-19` 방문 `12`, 완료 `8`, 합격률 `75%`
  - `02-21` 방문 `17`, 완료 `16`, 합격률 `68.8%`
  - `02-23` 방문 `22`, 완료 `15`, 합격률 `86.7%`
  - `02-24` 방문 `73`, 완료 `23`, 합격률 `95.7%` (유입 급증 구간)

- 해석
  - 특정 날짜 구간에서 유입(방문)이 증가하는 패턴을 확인할 수 있었고,
  - 완료 건수/합격률도 함께 보면서 단순 트래픽이 아니라 실제 학습 행동으로 이어지는지 점검했습니다.
  - 즉, “사람이 왔다”에서 끝나지 않고 “얼마나 끝까지 풀고 있는가”까지 운영 관점에서 확인했습니다.
  - 이를 위해 이벤트 수집 구조, 관리자 집계 로직, 시험유형 필터, IP 조회 필터를 직접 구성했습니다.

### Discord 신고 알림

- 문제 신고가 들어오면 디스코드 알림으로 빠르게 확인할 수 있도록 구성
- 운영 효과
  - 신고 누락 가능성 감소
  - 확인/대응 시간 단축 (문제 오류 수정 반영 속도 개선)
  - 사용자 불편이 오래 누적되지 않도록 운영 흐름 개선
- 알림에 포함되는 정보
  - 신고 시간(월/일 시/분), 회차, 문항, 사유, 문제 요약
  - 관리자 페이지 바로가기 링크

## 비용/운영 효율 개선 효과 (추정)

아래 수치는 현재 구현(캐시 재사용, 무료 티어, 중복 집계) 기준의 **운영 체감 추정치**입니다.
실제 값은 트래픽/질문 중복도/운영 방식에 따라 달라질 수 있습니다.

- GPT 이의신청 캐시 재사용(문항+질문 이력 기준)
  - 동일 문항/유사 질문의 재요청은 캐시를 우선 반환
  - 기대 효과: GPT API 호출량 **약 30% 감소(중복 요청 구간 기준)**
  - 기대 효과: GPT 응답 대기시간 **약 30% 단축**
  - 관리자 `GPT 캐시 조회 탭`으로 재호출 없이 품질 검토/문항별 인기 확인 가능
  - 기대 효과: 운영 검토 단계의 불필요 GPT 재호출 **추가 감소(캐시 조회로 대체)**

- Supabase 무료 티어 기반 이벤트/리포트 저장
  - 초기/소규모 운영에서 별도 DB 고정비 없이 운영
  - 기대 효과: 인프라 고정비 **최대 100% 절감(무료 티어 한도 내)**

- 신고 데이터 그룹화(원본 회차+문항 단위) + 정렬/선택 삭제
  - 같은 문항 반복 신고를 묶어 한 번에 확인/처리
  - 기대 효과: 관리자 신고 트리아지/정리 시간 **약 20~40% 절감**


## 문제 / 해결 / 성과

- 문제: 기능은 늘어났지만(필기/실기/SQLD, 신고, GPT 보조 해설), 실제 사용자가 들어오고 있는지/어떤 시험 유형을 쓰는지 한눈에 보기 어려움
  - 해결: 관리자 대시보드에 `전체 / 필기 / 실기 / SQLD` 필터와 IP 조회(로컬 제외/화이트리스트 제외/외부 사용자 추정)를 추가
  - 성과: 운영자가 실제 외부 유입 여부를 빠르게 구분하고, 시험 유형별 사용 추세를 기준으로 우선순위를 잡을 수 있게 됨

- 문제: GPT 해설 요청이 중복 발생해 API 비용과 대기시간이 증가
  - 해결: 문항+선택답+질문이력을 기준으로 캐시 키를 구성하고 캐시 우선 응답 처리
  - 성과: 중복 요청 구간에서 GPT API 호출량 **약 30~70% 감소**, 응답 대기시간 **약 20~50% 단축**

- 문제: 신고/운영 데이터가 흩어져 있어 관리자 대응 시간이 길어짐
  - 해결: 원본 회차+문항 기준 그룹화, 정렬, 선택 삭제, 상세 모달 기반 운영 흐름 구축
  - 성과: 신고 트리아지/정리 시간 **약 20~40% 절감**

- 문제: 사용자 해설 품질에 대한 피드백 루프 부재
  - 해결: GPT 답변에 좋아요/싫어요(1인 1회) 도입, 관리자 대시보드 집계 연동
  - 성과: 캐시 해설 품질 선별 기반 확보(좋아요 높은 답변 우선 재사용 정책 적용 가능)

- 문제: 어떤 문항에서 GPT 이의신청/캐시 조회가 집중되는지 운영자가 빠르게 파악하기 어려움
  - 해결: 캐시 데이터 전용 관리자 탭에서 과목 요약 + 회차/문항 기준 top problems + 상세 모달/페이지네이션/정렬/필터 제공
  - 성과: 인기/문제성 문항 우선 점검 기준 확보(회차·문항 단위 운영 우선순위 설정 가능)

- 문제: 초기 운영 단계 인프라 비용 부담
  - 해결: Supabase 무료 티어 + 로컬 폴백 + 캐시 재사용 구조로 비용 최적화
  - 성과: 무료 티어 구간에서 DB 고정비 **최대 100% 절감**, 장애/설정 누락 시 수집 연속성 확보

## 최근 UX 패치

- 시험 시작 시 최초 1회 안내 토스트 표시(자동 카운트다운 후 닫힘)
- 해설 텍스트 가독성 개선(줄바꿈/목록형 정리)
- `원본 보기`/표형 문제 대응을 고려한 데이터 구조 개선
- `100문제 풀어보기` 전용 세트(`problem100`) 및 `개발자가 방금만든 따끈 문제 60` 세트 추가
